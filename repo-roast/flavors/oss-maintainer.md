# OSS Maintainer Flavor — 开源维护者视角

## 权重

工程化 1.3x · 可读性 1.3x · 架构 0.8x · 安全 0.8x · 性能 0.8x

五个固定审查维度：`architecture`、`security`、`performance`、`readability`、`engineering`。

开源项目的第一用户是贡献者。没人看得懂 = 没人贡献。文档和 CONTRIBUTING 指南是第一优先级。

## 审查立场

以贡献者上手成本、公共接口可读性、治理和长期维护为优先。评价问题时考虑它是否阻碍新的贡献者理解、运行或安全修改项目。

## 与 Tone 的边界

本 Flavor 只决定五维权重、关注点、排序和报告组织，不决定表达攻击性。它可与 `professional`、`sharp`、`savage` 任意组合。

## 特别关注

- CONTRIBUTING.md 是否存在、是否清晰
- Issue 模板有没有、PR 模板有没有
- 新人 onboarding 体验：clone → build → run 几步？
- 项目治理：LICENSE 有没有、CODE_OF_CONDUCT 有没有
- 社区响应：最近的 issue/PR 有没有得到回复
- 架构文档：有没有 ARCHITECTURE.md 或设计决策记录

## 报告调整

- "社区友好度"章节替代部分工程化内容
- 每个问题附带 first-good-issue 级别的改进建议
- 增加"新人上手路径"建议

## 立场禁区

- 不把维护资源不足推断为态度问题
- 不把可选治理实践当作功能正确性的必要条件
