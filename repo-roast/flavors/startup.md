# Startup Flavor — 硅谷创业公司视角

## 权重

工程化 1.3x · 可读性 1.2x · 架构 1.0x · 安全 0.8x · 性能 0.7x

五个固定审查维度：`architecture`、`security`、`performance`、`readability`、`engineering`。

创业公司关注"能不能上线"和"技术债能不能还"。功能先跑通，但要清楚欠了多少债。

## 审查立场

以交付速度、上线阻塞项和技术债可控性为优先。允许非关键能力暂时不完美，但必须把核心流程风险和债务成本记录清楚。

## 与 Tone 的边界

本 Flavor 只决定五维权重、关注点、宽容项、排序和报告组织，不决定表达攻击性。它可与 `professional`、`sharp`、`savage` 任意组合。

## 特别关注

- MVP 是否可以交付
- 技术债是否可控（可以欠，但要知道欠在哪、欠多少）
- 上线前的 blockers：监控有没有？日志够不够？错误追踪接了吗？
- 上手成本：新人多久能改第一行代码？
- 有没有过度抽象（为"以后"写的代码现在用不上）

## 宽容项

- 测试覆盖率低（早期可以接受，但核心流程要有）
- CI 不完善（但 build + deploy 自动化要有）
- 文档不全（但 README 必须包含 how to run）
- 性能没有极致优化（但不要有明显的 N+1 或内存泄漏）

## 报告调整

- 工程化和架构章节提前
- 把问题分成"上线前必修"和"可以边跑边修"两档

## 立场禁区

- 不以企业级完整度作为早期项目的默认门槛
- 不在缺少证据时建议全面重构
