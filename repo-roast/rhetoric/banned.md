# 锐评禁区

本文件约束 `professional`、`sharp`、`savage` 三种 Tone。命中禁区的候选锐评必须被删除，报告保留原始技术内容。

## 禁止攻击对象

禁止评价或影射代码作者、贡献者、团队和用户的：

- 智力、常识、教育程度或理解能力；
- 人格、品德、心理状态或价值；
- 职业资格、薪资、级别或是否应该转行；
- 年龄、性别、地域、国籍、种族、宗教、健康等身份特征；
- 未经证实的动机，例如“偷懒”“故意”“刷简历”“不负责任”。

禁止建议任何人辞职、转行、放弃项目，或用服从、羞辱和持续施压换取修改。

## 禁止表达方式

- 空洞辱骂：只表达“烂”“蠢”“垃圾”，没有技术信息。
- 灾难膨胀：把 `low` 或 `info` 描述成系统必然崩溃、公司必然倒闭。
- 无证据定罪：没有调用路径或输入条件却断言漏洞可被利用。
- 动机推断：根据实现推断作者态度、目的或工作习惯。
- 重复 PUA：反复强调“不改就不配”“这都做不到”等服从性压力。
- 群体刻板印象：拿语言、框架、地区或从业群体作为攻击对象。
- 暴力、疾病、死亡或自残隐喻。
- 性暗示、仇恨表达和歧视性词汇。
- 用玩笑冲淡 Critical 安全问题、数据损坏或用户伤害风险。
- 每条 finding 都添加笑话，导致报告失去信息层级。

## 禁止修改事实

修辞不得：

- 添加 finding 中没有出现的模块、请求、用户或生产环境。
- 把“在特定输入下”改成“任何时候都会”。
- 把“维护成本”改成“运行故障”。
- 把“疑似”改成“已被利用”。
- 把建议改成没有依据的全面重构。
- 使用语料文本补齐缺失的 `impact` 或 `suggestion`。

## 报告级限制

- 同一模板 id 不得重复。
- 相同锐评句不得重复。
- 相邻两个问题不得使用相同修辞手法。
- 一句话总评中的玩笑不得在问题正文再次使用。
- Critical finding 默认不使用夸张类比；只有不弱化风险且不遮挡成立条件时才能使用克制表达。

## 安全替代

发现候选文本越界时，不尝试用同义词规避禁令。直接移除锐评句，保留：

```text
title + file:line + description + impact + suggestion
```

纯技术表达始终是合法回退结果。
