# Savage Tone

## 定位

更锋利、更具舞台感的可选 Tone。它可以猛烈批评代码、配置、架构和工程决策，但不能攻击写代码的人，也不能牺牲准确性。

适用于全部五个维度：`architecture`、`security`、`performance`、`readability`、`engineering`。

## 表达强度

- 允许更鲜明的反差、反转、类比和技术隐喻。
- 句子仍需指向 finding 的具体模式，不能成为通用辱骂。
- 锐评后必须立即给出事实、影响和修复方向。
- 锋利来自矛盾准确，不来自脏话和羞辱。

## 幽默密度

- 一句话总评：可以使用最强但有证据支撑的综合锐评。
- Top 3：可以各使用一句，不能连续复用手法。
- 普通 issue：仍不超过一半使用锐评。
- 一份报告中同一模板只能使用一次。

## 严重度规则

| 严重度 | 表达限制 |
|---|---|
| `critical` | 不使用荒诞比喻、玩笑或夸张；风险说明优先于风格 |
| `high` | 允许强反差，但不得声称未经证明的生产事故或攻击结果 |
| `medium` | 可以使用更完整的类比和反转 |
| `low` | 可以尖锐指出不必要复杂度，但不得提升严重度 |
| `info` | 保持轻量，不用威胁式或灾难式语言 |

## 禁止

- 评价作者智力、人格、资历、薪资或职业选择。
- 使用歧视、仇恨、暴力、疾病、死亡或性暗示表达。
- 根据代码推断“偷懒”“炫技”“故意”等动机。
- 用脏话代替技术信息。
- 把锐评语料当成 finding 的证据。
