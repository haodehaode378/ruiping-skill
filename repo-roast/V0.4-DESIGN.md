# Repo-Roast v0.4.0：证据驱动的锐评修辞引擎

## 目标

v0.4.0 将 Repo-Roast 从“带锐评语气的代码审查 Skill”升级为“证据驱动的技术锐评系统”。报告可以有鲜明表达，但任何修辞都必须建立在已经验证的技术事实之上。

系统固定审查以下五个维度，名称不得改写或扩展：

- `architecture`
- `security`
- `performance`
- `readability`
- `engineering`

版本成功的判断标准不是“更会骂”，而是同一条 finding 在改变表达强度后，严重度、证据、事实、影响和建议保持不变。

## 非目标

- 不将 PUA、羞辱或人身攻击引入代码审查。
- 不让修辞模板参与漏洞、缺陷或严重度判断。
- 不为匹配一句锐评而补充未经验证的事实。
- 不新增第六个审查维度。
- 不把 Repo-Roast 改造成 lint、漏洞利用器或自动修复工具。
- 不在 v0.4.0 引入与 Markdown/JSON 协议无关的运行时或框架。

## 三层数据流

```text
Scout 项目画像
      │
      ▼
事实层：五维 Deep Dive 输出中性 Review JSON
      │  已验证的 file + line + 技术事实
      ▼
判断层：Editor 验证、去重、定级、排序、计算权重
      │  不改变原始 finding 字段
      ▼
表达层：Rhetoric Engine 按 Flavor + Tone 匹配修辞
      │  无可靠匹配时安全回退
      ▼
包含事实、影响、建议和可选锐评句的 Markdown 报告
```

### 事实层

Scout 只建立项目画像和统一审查范围。五个 Deep Dive Agent 只判断问题是否成立，并输出严格 JSON。

Deep Dive 必须：

- 使用仓库内相对路径和真实正整数行号。
- 将可观察事实、实际影响和修改建议分开表达。
- 对问题分配固定五维之一以及稳定的 `pattern`。
- 在证据不足时不输出正式 issue。
- 保持中性，不读取锐评语料，不生成戏剧化、侮辱性或修辞性文本。

### 判断层

Editor 是技术判断结果的消费者，而不是新的事实发现者。它负责：

- 全量验证每条 issue 的文件和行号。
- 删除证据无效的正式 issue。
- 合并重复 finding，并保留更高但有证据支持的严重度。
- 根据 Flavor 权重计算总分并排序。
- 区分请求、完成、失败和跳过的维度。
- 生成报告所需的聚合信息。

Editor 不得改变 Deep Dive 已输出的 `severity`、`file`、`line`、`description`、`impact` 或 `suggestion`。如果聚合时发现字段矛盾，应舍弃或明确标记该 finding，而不是自行补写事实。

### 表达层

Rhetoric Engine 只在判断层完成后运行。它根据以下顺序选择一条可选表达：

```text
dimension → pattern → severity → tone → 去除本报告已使用项
```

每个正式问题最多添加一句主要锐评。没有可靠匹配、证据不足、表达可能弱化风险或会造成重复时，必须回退到不带锐评的技术描述。

删除报告中的全部锐评句后，技术事实、实际影响和修改建议必须仍然完整成立。

## Flavor 与 Tone

### Flavor：从什么立场审查

Flavor 决定五维权重、问题排序倾向、关注点、宽容项和报告组织。支持：

- `default`
- `google`
- `startup`
- `oss-maintainer`

每个 Flavor 都必须显式引用 `architecture`、`security`、`performance`、`readability`、`engineering` 五个维度。Flavor 不负责控制语言攻击性。

### Tone：用多锋利的语言表达

Tone 只改变措辞，不改变 finding。支持：

- `professional`：冷静、直接、正式，不依赖幽默。
- `sharp`：默认值；允许建设性的类比、反差、反问和冷幽默。
- `savage`：允许更强讽刺，但仍只能批评代码、配置、架构和工程决策。

任何 Tone 都禁止评价作者的人格、智力、职业资格和动机。核心规则是：**Roast the code, never the coder.**

Flavor 与 Tone 可以自由组合。例如，`startup + professional` 强调上线风险并使用正式措辞，`google + savage` 强调规范和可维护性并使用更锋利的表达。

## Issue 证据协议

每个正式 issue 必须包含：

- `file`：仓库内非空相对路径，不得是绝对路径或 URL。
- `line`：大于等于 1 且不超过文件实际行数的整数。
- `description`：在引用位置可观察的技术事实。
- `impact`：该事实造成的具体后果或风险。
- `suggestion`：可执行的修改方向。

Editor 必须逐条验证全部正式 issue，不能以抽查替代全量验证。文件不存在、行号越界或事实无法定位的 finding 不得进入正式问题列表，也不得通过降低严重度继续保留为确定性结论。

### 缺失型问题

“没有测试”“没有 CI”“没有 CHANGELOG”等缺失型事实通常没有天然的文件与行号。此类事实应进入：

- Scout 或维度结果的 `stats`；
- 维度 `summary`；
- 明确标注为仓库级建议的处方。

除非存在能够直接证明配置缺口或错误声明的真实文件行，否则不得为缺失型事实伪造 `file` 和 `line`，也不得将其包装成正式 issue。

## 修辞安全边界

修辞必须满足：

1. 技术事实先于修辞成立。
2. 修辞不能增加原始 finding 中不存在的信息。
3. 修辞不能提升或降低问题严重度。
4. 修辞不能取代 `description`、`impact` 或 `suggestion`。
5. Critical 问题优先清楚说明风险，不强行玩梗。
6. 无匹配时输出纯技术描述。
7. 同一报告避免重复句式和连续使用相同修辞手法。
8. 只批评代码和工程决策，不攻击人。

## 失败与安全回退

| 失败场景 | 处理方式 |
|---|---|
| Deep Dive JSON 无法解析 | 标记对应请求维度失败，不交给修辞层 |
| issue 文件不存在或行号越界 | 从正式问题列表删除 |
| `pattern` 缺失或不受支持 | 保留技术 finding，不添加锐评 |
| Tone 文件缺失 | 回退到 `sharp`；`sharp` 也缺失则输出纯技术描述 |
| Flavor 文件缺失 | 回退到 `default`，不改变 finding |
| 修辞模板与严重度不匹配 | 不使用该模板 |
| 修辞命中禁止措辞 | 删除修辞句，保留技术内容 |
| 某请求维度超时 | 标记为失败；不得伪装成用户跳过 |
| 所有请求维度失败 | 输出无法评定的错误报告，不生成虚构评分 |

安全回退的基本策略是“宁可少一句锐评，也不多一个未经证实的结论”。

## v0.3 到 v0.4 的兼容策略

v0.4 保留现有五维评分、Flavor 名称和 Markdown 报告总体结构，但升级内部数据契约。

| v0.3 字段/行为 | v0.4 迁移 |
|---|---|
| issue 没有 `title` | Deep Dive 必须输出事实型短标题 |
| issue 没有 `impact`，模板使用 `risk` | 统一为 `impact`，移除重复的 `risk` 语义 |
| issue 没有 `pattern` | Deep Dive 输出稳定 kebab-case 模式；无法判断时使用受控通用模式，不得猜测事实 |
| highlight 没有 `title` | 统一契约并使模板字段与输出一致 |
| Flavor 同时承担语气职责 | Flavor 保留立场与权重，新增独立 Tone |
| Editor 抽查 2–3 条证据 | 改为全量验证正式 issue |
| 完成维度固定显示 `/5` | 改为请求、完成、失败、跳过四种状态和准确分母 |
| 无匹配语言材料 | 明确回退到纯技术描述 |

旧版 Review JSON 不作为 v0.4 的静默兼容输入。若字段缺失，Editor 应将其视为契约错误并标记对应维度失败，避免临场补写 `title`、`impact` 或 `pattern` 产生幻觉。

## v0.4.0 验收不变量

- 五个维度名称在 Schema、Prompt、Flavor、Tone 和模板中一致。
- Deep Dive 永远不读取锐评语料。
- Editor 永远不改变原始技术事实。
- 同一 finding 在三种 Tone 下的事实字段完全一致。
- 每条正式 issue 的文件和行号都经过验证。
- 修辞模板不能独立构成新的技术结论。
- Flavor 和 Tone 可以自由组合。
- 未匹配修辞时报告仍能正常生成。
- 所有协议文件保持 UTF-8，结构化文件保持标准 JSON。
